⽬前流⾏⼏种备份⽅式包括逻辑备份、物理备份和双机热备份。

逻辑备份

使⽤mysql⾃带的mysqldump⼯具进⾏备份。备份成sql⽂件形式。
优点 ：最⼤好处是能够与正在运⾏的mysql⾃动协同⼯作， 在运⾏期间可以确保备份是当时的点，它会⾃动将对应操作的表锁定，不允许其他⽤户修 改(只能访问)。可能会阻⽌修改操作。sql⽂件通⽤⽅便移植。
缺点：备份的速度⽐较慢。如果是数据量很多的时候。就很耗时间。如果数据库服务器处 在提供给⽤户服务状态，在这段⻓时间操作过程中，意味着要锁定表(⼀般是读锁定，只能 读不能写⼊数据)。那么服务就会影响的。

物理备份

直接拷⻉mysql的数据⽬录。 直接拷⻉只适⽤于myisam类型的表。这种类型的表是与机器独⽴的。但实际情况是，你 设计数据库的时候不可能全部使⽤myisam类型表。你也不可能：因为myisam类型表与机 器独⽴，⽅便移植，于是就选择这种表，这并不是选择它的理由。 缺点：你不能去操作正在运⾏的mysql服务器(在拷⻉的过程中有⽤户通过应⽤程序访问更 新数据，这样就⽆法备份当时的数据)可能⽆法移植到其他机器上去。 更多的情况是，你会根据业务特点(⽐如你需要⽀持事务机制就必须使⽤innodb)，查询速 度和服务性能来选择表类型的。 必须保证表不被使⽤中。 如果服务器在你正在拷⻉⼀个表时(某个⽤户在进⾏update、insert操作)改变它，拷⻉的数 据就失去了备份的意义(没法准确还原到当时的点)。 保证你的拷⻉完整性的最好⽅法是：关闭服务器，拷⻉⽂件，然后重启服务器。 或者是，要锁定对应的表(对前端⽤户造成访问问题)。

双机热备份

mysql数据库没有增量备份的机制。当数据量太⼤的时候备份是⼀个很⼤的问题。还好 mysql数据库提供了⼀种主从备份的机制(也就是双机热备) 优点：适合数据量⼤的时候。搭建多台数据库服务器，进⾏主从复制。 主从复制经常遇到的问题就是，如何保证数据不堵塞，不延迟

主从同步不能代替数据库备份

举例说明，如果我们在主数据库上误删了⼀个table，那么从数据库也会执⾏相应的操作， 如果没有备份，这就会导致删掉的数据就⽆法找回。（即使回滚⼆进制⽇志也不⼀定能找 回，因为⼆进制⽇志的保存是有时间限制的，并且回滚⼆进制⽇志⾮常复杂）。 因此数据库的完全备份、增量备份是⾄关重要的。

主从备份策略

  1. 每天进⾏数据库备份。
    mysqldump --master-data --single-transaction -R --databases [db1] [db2] [db3] | gzip -9 - | pv >all-db-with-master-data-$(date +%Y%m%d).sql.gz
  2. 将每天的备份同步到从服务器上，以便随时进⾏从数据库的搭建。
  3. 数据库的恢复
  • 恢复主数据库。
    将master-data⽣产的从数据库信息注释掉，然后恢复即可。
  • 恢复从数据库。
    使⽤备份的数据⽂件，恢复从数据库，然后启动slave即可。

备份过程中遇到的坑（视图）

有些时候我们只需要对表结构进⾏备份。在⼀次数据恢复过程中，发现⼀些视图丢失了， 还有⼀些视图被当成table备份了；经过⼀番查资料和动⼿操作，总结了以下步骤，希望能 帮助⼤家（环境是linux+mysql）：

  1. 在需要备份的数据库服务器上⽤命令 mysqldump -u root -p –skip-opt –skip-comments –complete-insert ‘your db name’ > ‘your file name’这个命令会把你的数据库结构和相关数 据都导出来，如果你只要数据库结构,可以⽤以下命令：mysqldump -u root -p -d –skip-opt –skip-comments –complete-insert ‘your db name’ > ‘your file name’;
  2. 把备份后的⽂件通过scp等安全传送⼯具传送到相关的数据库服务器上，我⼀般使⽤ winscp；
  3. 在服务上⽤mysql -u ‘db user name’ -p 进⼊服务器；
  4. create database ‘your db name’ character set utf8;
  5. set names utf8;
    1. source ‘your file name’;
    2. 系统会提⽰成功； 这个时候⽤select * from ‘your view name’;发现数据显⽰出来了。
